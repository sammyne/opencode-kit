请按以下 5 个阶段，一步一步完成我的需求。

## 阶段 0：确定规划目录名称（自动执行）
1. 分析我的原始需求："$ARGUMENTS"。
2. 提取核心意图，自动生成一个**简洁、英文的文件夹名称**（使用 kebab-case 命名法，例如 `order-excel-export`）。
   - 如果描述中包含英文关键词，优先使用。
   - 如果全是中文，自动翻译核心功能为英文。
3. 定义本次规划的根路径为：`.opencode/plans/{文件夹名称}/`。
4. **无需等待确认，直接进入阶段 1。**

## 阶段 1：生成需求文档
- 在确定的根路径下创建 `requirements.md`。
- **文件格式要求：**
  - **文件的第一行必须是一级标题 `# 需求`**。
  - 在该标题下撰写详细内容。
- 内容至少包含：
  - 目标与背景
  - 功能需求列表
  - 非功能需求（性能、安全、兼容性等）
  - 边界与不做事项
  - 假设与约束
- 生成完成后，停下来并明确提醒我："需求文档已生成在 `{根路径}/requirements.md`，请确认内容。"
- （这是第一次交互暂停点）

## 阶段 2：等待用户确认（需求文档）
- 仅根据我的反馈修改 `requirements.md` 的内容。
- 收到我"确认"或"继续"的明确回复后，再进入下一阶段。

## 阶段 3：生成 TODO 列表
- 基于确认后的需求文档，在确定的根路径下创建 `todo.md`。
- **文件格式要求：**
  - **文件的第一行必须是一级标题 `# TODO`**。
  - 在该标题下输出结构化的任务列表。
- 内容至少包含：
  - 每个条目的简要描述
  - 优先级（P0/P1/P2）
  - 预估风险或注意点
- 生成完成后，停下来并明确提醒我："TODO 列表已生成在 `{根路径}/todo.md`，请确认是否进入编码阶段。"
- （这是第二次交互暂停点）

## 阶段 4：等待用户确认（TODO）
- 仅根据我的反馈调整 `todo.md` 的内容。
- 收到我"确认"或"开始编码"的明确回复后，再进入下一阶段。

## 阶段 5：编码实现
- 在此阶段之前，请提醒我切换到 Build 模式（例如："现在请切换到 Build 模式，我将根据 `{根路径}/todo.md` 开始编码。"）。
- 切换到 Build 模式后，按 `{根路径}/todo.md` 中的列表顺序执行编码。
- **每完成一个 TODO 项后：**
  1. 更新 `{根路径}/todo.md`，在对应条目前打勾或补充说明。
  2. **立即输出一条提醒**（例如："✅ 已完成：[任务描述]"）。
  3. **无需等待用户确认，直接继续下一个 TODO 项。**
- 生成代码时遵循项目既有代码风格和结构，必要时询问我文件名、路径等。
- 如果遇到需要我决策的地方，停下来询问，不要擅自决定。

**重要约束：**
- 在阶段 0–4 内（Plan 模式下），不要执行任何对业务代码的 write/edit/bash 操作，只输出文档/计划和文本建议。
- 所有的规划文件必须存放在阶段 0 自动确定的 `.opencode/plans/{文件夹名称}/` 目录下。
